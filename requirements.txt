RailEasy

Purpose
Build a train ticket booking web application where users can search trains between stations, choose a travel class,
select seats, and confirm a booking. Admins manage trains and train schedules. The focus is on foundational
full-stack skills - React/Angular, Spring Boot, REST, PostgreSQL, Git, and basic testing - aligned with the Foundation
skill matrix.

1) About Us
    At RailEasy, we believe train travel should be booked in minutes, not hours. This Foundation version delivers a clean train
    search, class selection, and booking experience with essential features only.

2) Problem Statement
    Train travellers need a simple way to find trains between stations, view class availability, and confirm a ticket without
    navigating complex government portals.
    Goal: Deliver a minimal yet complete train booking flow demonstrating full-stack fundamentals.

3) Roles
    1. Passenger/User - Can search trains, view class availability, book and cancel tickets.
    2. Admin - Can manage trains and schedules.
    Note: Simple isAdmin boolean flag on the User entity.
    Role Flag / Field Approach
        • User entity: boolean isAdmin; (defaults to false on registration)
        • Seed one admin user with isAdmin = true.
        • Backend: check isAdmin before allowing train/schedule CRUD.
        • Frontend: show Admin menu only when isAdmin is true in user context.

4) In-Scope Features
    Authentication (Basic)
    • Register, Login, Logout.
    • Simple password hashing (BCrypt).
    Train Search
        • Search form: From Station, To Station, Date.
        • Results: train name, number, departure, arrival, available classes with seat counts and fare.
    Class & Seat Selection
        • Select a travel class (SLEEPER / AC_3 / AC_2).
        • Simple seat grid for selected class (8 rows x 8 seats = 64 per class).
        • Seats shown as Available or Booked.
        • Select 1-4 seats and confirm.
    Booking & My Tickets
        • Confirm booking -> PNR number generated (UUID first 8 chars).
        • My Tickets page: list of bookings with PNR, train, date, class, seats, status.
        • Cancel a ticket (seats freed, status = CANCELLED).
    Admin - Train & Schedule Management
        • CRUD for trains (trainNumber, trainName, totalSeatsPerClass).
        • CRUD for schedules (trainId, fromStation, toStation, departureTime, arrivalTime, journeyDate, farePerClass).

5) User Flows (Happy Path)
    1. Search: User enters From Station, To Station, Date -> sees train list.
    2. Select Class: Clicks a train -> sees class options with fare and availability.
    3. Seat Map: Selects class -> sees seat grid -> picks seats.
    4. Auth: If not logged in -> prompted to register/login.
    5. Confirm: Reviews booking -> confirms -> sees PNR number.
    6. My Tickets: Views all tickets -> can cancel upcoming journeys.
    7. Admin: Logs in -> manages trains and schedules.

7) Architecture
    • Frontend: React (Hooks + Context API) or Angular (Services + Components). Routing enabled.
    • Backend: Spring Boot (REST Controllers, Services, Spring Data JPA).
    • Database: PostgreSQL (Dev may use H2 for quick start).
    • Auth: Stateless JWT or simple session cookie (choose one; document trade-offs).
    • State: Selected class and seats held in frontend state.
    • Deployment: Localhost

9) Data Model
    Entities & Relationships
        • Train (1) -- (M) Schedule
        • User (1) -- (M) Booking
        • Schedule (1) -- (M) Booking
        • Booking stores: travelClass (ENUM), seatNumbers (CSV string e.g. '1A,1B'), pnrNumber, status
    Note: Store booked seats as a CSV string per booking. Derive availability by counting booked seats vs totalSeatsPerClass.

10) API Outline
    Auth
        • POST /api/auth/register - {email, password, name} -> 201
        • POST /api/auth/login -> 200 {token}
        • POST /api/auth/logout -> 204
    Trains & Schedules
        • GET /api/schedules?from=&to;=&date;= - search schedules with class availability
        • GET /api/schedules/{id}/seats?class= - booked seat numbers for class
        • POST /api/schedules (admin)
        • PUT /api/schedules/{id} (admin)
        • DELETE /api/schedules/{id} (admin)
    Trains (Admin)
        • GET /api/trains
        • POST /api/trains
        • PUT /api/trains/{id}
        • DELETE /api/trains/{id}
    Bookings
        • POST /api/bookings - {scheduleId, travelClass, seatNumbers[]} -> creates booking + PNR
        • GET /api/bookings/mine - user's tickets
        • PUT /api/bookings/{id}/cancel - cancel ticket

Error Format (Consistent)
    {
        "timestamp": "2025-09-12T12:00:00Z",
        "path": "/api/...",
        "error": "VALIDATION_ERROR",
        "message": "Field is required"
    }

11) NFRs & SLAs
    • Performance:
        • API response <= 2 s under light load (1-5 concurrent users locally).
        • UI renders initial content within 3 s on localhost.
    • Security:
        • Passwords hashed (BCrypt); JWT/session token stored securely.
        • Basic input validation & server-side checks.
    • Scalability/Concurrency:
        • Test with 5-10 concurrent requests locally. Simulate using Postman Runner or multiple browser tabs. No JMeter or
            load-testing tools required at this level.
    • Error Handling:
        • User-friendly error toasts/messages on UI; structured error JSON on API.
    • Logging:
        • Console logs; simple request/response logs (no external logging infra).
    • Documentation:
        • README.md with setup steps, seed data, and run commands.
        • API contracts documented with Swagger/OpenAPI.

12) Testing
    • Backend Unit Tests
        • Services & Repos with JUnit5 + Mockito
        • Target coverage: >=60% (method-level)
    • API Testing
        • Manual via Postman or Swagger.
    • Frontend
        • Optional unit tests for 1-2 key components (e.g., Booking form).
    • No automated regression/e2e required.

13) Engineering PoV
    • Code Quality:
        • Follows SOLID basics where practical (e.g., service layer, DTOs).
        • No critical lint errors (ESLint/TSLint).
    • Branching:
        • main (stable), feature/* branches; PR reviews by peers/mentor.
    • Coverage:
        • Aim 60-70% backend, frontend minimal.
    • Dependencies:
        • Keep list short; beware of transitive vulnerabilities.
    • Docs:
        • API endpoints documented; Postman collection included.

15) Sample Acceptance Criteria
    Train Search
        • Given I search Chennai -> Mumbai on a seeded date, I see at least 2 trains.
        • Each result shows available classes with seat count and fare.
    Seat Selection
        • Given I select AC_3 class, I see an 8x8 grid with booked seats highlighted.
        • I cannot select an already-booked seat.
    Booking
        • After confirming, I receive a PNR number on the confirmation screen.
        • Cancelling a ticket changes status to CANCELLED and frees the seats.
    Admin
        • Only admin can access /admin.
        • Admin adds a train and it is selectable when creating a new schedule.

18) Sample User Stories
    • As a passenger, I want to search trains by station and date so I can plan my journey.
    • As a passenger, I want to choose a travel class so I can book within my budget.
    • As a passenger, I want to see available seats so I can pick my preferred berth.
    • As a passenger, I want a PNR number after booking so I can reference my ticket.
